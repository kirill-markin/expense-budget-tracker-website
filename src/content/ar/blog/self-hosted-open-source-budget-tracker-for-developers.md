إذا كنت مطورًا، فالأرجح أن بياناتك المالية مخزنة اليوم على خادم لا تملكه أنت. معظم تطبيقات تتبع الميزانية والمصروفات مثل Mint وYNAB وCopilot وLunch Money تحفظ معاملاتك، وأرصدة حساباتك، وأنماط إنفاقك في سحابة الشركة نفسها. أنت تراهن فقط على ألا تتعرض للاختراق، وألا تبيع بياناتك، وألا تختفي الخدمة فجأة كما حدث مع Mint في عام 2024.

إذا كنت مرتاحًا للعمل مع Docker وPostgres، فهناك خيار أوضح وأفضل: استضف متتبّع ميزانية مفتوح المصدر بنفسك، واحتفظ ببياناتك داخل بنيتك أنت.

## أداة مفتوحة المصدر تديرها على بنيتك أنت

[Expense Budget Tracker](https://github.com/kirill-markin/expense-budget-tracker) نظام مفتوح المصدر بالكامل لتتبع المصروفات والميزانية، مبني على Postgres. تستنسخ المستودع، وتشغّل `make up`، فتحصل على تطبيق يعمل على `localhost:3000` مع قاعدة بيانات فعلية أنت من يتحكم بها.

لا تحتاج إلى إنشاء حساب، ولا تغادر بياناتك جهازك أو خادمك، ولا توجد رسوم اشتراك. والمشروع مرخّص تحت MIT، ما يعني أنك تستطيع نسخه وتعديله واستخدامه بالطريقة التي تناسبك.

![جدول ميزانية يعرض الأرقام الفعلية السابقة وتتبع الشهر الحالي وتوقع الأشهر المقبلة حسب الفئة](/blog/budget-view-example.jpg)

البنية التقنية مباشرة وواضحة: Next.js لواجهة الويب، وPostgres 18 للتخزين، وعامل مكتوب بـ TypeScript لجلب أسعار الصرف اليومية. وكل ذلك يعمل داخل حاويات Docker عبر ملف `docker-compose.yml` واحد.

## استضافة ذاتية عبر Docker أو نشر على AWS

يوفر المستودع خيارين جاهزين للنشر من البداية:

**Docker Compose محلي**: أربع خطوات قصيرة ويصبح التطبيق جاهزًا للعمل:

```bash
git clone https://github.com/kirill-markin/expense-budget-tracker.git
cd expense-budget-tracker
open -a Docker   # شغّل Docker إذا لم يكن يعمل
make up          # قاعدة البيانات + الترحيلات + الواجهة + العامل
```

بعد ذلك افتح `http://localhost:3000` وابدأ بإدخال معاملاتك. وتبقى بيانات Postgres محفوظة داخل وحدة تخزين Docker. هذا كل ما يلزم للإعداد المحلي.

**AWS CDK**: نشر إنتاجي كامل عبر أمر واحد:

```bash
bash scripts/bootstrap.sh --region eu-central-1
```

سيُنشئ هذا ECS Fargate وRDS Postgres وALB مع HTTPS وCognito للمصادقة وWAF وCloudWatch للمراقبة ونسخًا احتياطية تلقائية وCI/CD عبر GitHub Actions. التكلفة التقديرية تقارب 50 دولارًا شهريًا، وفي المقابل تحصل على بنية تحتية بمستوى إنتاجي تملكها وتتحكم بها بالكامل. ويشرح [دليل النشر](https://github.com/kirill-markin/expense-budget-tracker/blob/main/infra/aws/README.md) كل خطوة بالتفصيل، من إنشاء حساب AWS إلى إعداد DNS في Cloudflare.

وبما أن النظام في جوهره Postgres مع Docker، فيمكنك استضافته تقريبًا في أي مكان آخر أيضًا: DigitalOcean أو Hetzner أو Raspberry Pi في المنزل أو عنقود Kubernetes في شركتك. أي بيئة تشغّل Docker وPostgres ستشغّل هذا المشروع.

## واجهة SQL API للوصول البرمجي

معظم تطبيقات الميزانية تكتفي بواجهة ويب. هذا المشروع يقدّم أكثر من ذلك: **واجهة SQL API** عبر HTTP، أي نقطة `POST /v1/sql` تستقبل أوامر SQL مباشرة وتعيد النتائج بصيغة JSON.

```bash
curl -X POST https://api.your-domain.com/v1/sql \
  -H "Authorization: ApiKey ebta_a7Bk9mNp..." \
  -H "X-Workspace-Id: workspace-id" \
  -H "Content-Type: application/json" \
  -d '{"sql": "SELECT category, SUM(amount) AS total FROM ledger_entries WHERE kind = '\''spend'\'' AND ts >= DATE_TRUNC('\''month'\'', CURRENT_DATE) GROUP BY category ORDER BY total"}'
```

تنشئ مفتاح API من صفحة الإعدادات، وتحدد معرّف مساحة العمل المستهدفة، وبعدها يستطيع أي عميل HTTP الاستعلام عن بياناتك. لا توجد هنا طبقات معقدة أو تجريدات زائدة: لا GraphQL، ولا ORM، ولا SDK تحتاج إلى تعلّمه. SQL يدخل، وJSON يخرج.

كما أن نموذج الأمان صارم وواضح: مفاتيح API تُخزَّن على شكل SHA-256 hashes من دون حفظ القيم الأصلية، والاستعلامات محصورة في `SELECT` و`INSERT` و`UPDATE` و`DELETE` فقط من دون DDL، وهناك مهلة تنفيذ قدرها 30 ثانية، وحد أقصى 100 صف في الاستجابة، وحد لمعدل الطلبات يبلغ 10 طلبات في الثانية لكل مفتاح. وكل الاستعلامات تمر عبر Postgres Row Level Security، وهو العزل نفسه الذي تستخدمه واجهة الويب، لذلك لا يستطيع أي مفتاح API الوصول إلا إلى بيانات مساحة العمل التابعة له.

## مصمم لوكلاء الذكاء الاصطناعي والنماذج اللغوية

وجود SQL API هو ما يجعل دمج الذكاء الاصطناعي عمليًا فعلًا في إدارة المال الشخصي، لأن الوكيل يحتاج إلى قراءة البيانات المالية وكتابتها مباشرة في قاعدة البيانات.

فكر في الطريقة المعتادة اليوم: ترفع لقطة شاشة لكشف حسابك إلى Claude أو ChatGPT، وتطلب منه تصنيف المصروفات، فيعطيك ملخصًا جيدًا. ثم تعود أنت وتنسخ الأرقام يدويًا إلى الأداة التي تستخدمها. هذا تدفق عمل متعب وقديم.

أما مع SQL API، فالوكيل لا يكتفي بالتحليل، بل **يكتب مباشرة إلى قاعدة بياناتك**. عندها يصبح التدفق كالتالي:

1. تضع كشفًا بنكيًا، سواء كان CSV أو PDF أو لقطة شاشة، داخل وكيل ذكاء اصطناعي.
2. يقرأ الوكيل كل معاملة، ويطابقها مع فئاتك الحالية، ثم ينفذ `INSERT` في جدول `ledger_entries`.
3. يراجع رصيد الحساب ويقارنه بالرقم الصادر من البنك.
4. تقضي أنت بضع دقائق في المراجعة بدل أن تمضي ساعة كاملة في الإدخال اليدوي.

وقد صُمم مخطط قاعدة البيانات ليتوافق مع هذا الاستخدام. هناك سبعة جداول مسطحة، بلا JSON متداخل، ومن دون عمليات ربط معقدة للمهام الأساسية. وجدول `ledger_entries` بسيط عمدًا: صف واحد لكل حركة حساب، وأسماء أعمدة واضحة وصريحة. لهذا يستطيع النموذج اللغوي كتابة أوامر `INSERT` صحيحة من المحاولة الأولى في أغلب الحالات.

ويضم Expense Budget Tracker أيضًا **دردشة ذكاء اصطناعي مدمجة** داخل واجهة الويب. ما عليك إلا توصيل مفتاح OpenAI أو Anthropic API لتحصل على مساعد يملك أداة `query_database`، ويمكنه تنفيذ `SELECT` و`INSERT` و`UPDATE` و`DELETE` مباشرة على Postgres لديك. ارفع لقطة شاشة من تطبيقك البنكي، وسيحلل الذكاء الاصطناعي المعاملات، ويطلب تأكيدك، ثم يضيفها. كما أنه يعمل وفق بروتوكول واضح: يكتشف فئاتك الموجودة أولًا، ويتحقق من التكرار، ويطابق الأرصدة، ولا يكتب أي شيء قبل موافقتك الصريحة.

وتدعم هذه الدردشة نماذج Claude من Anthropic ونماذج GPT من OpenAI. ويستخدم الطرفان أداة قاعدة البيانات نفسها وقواعد الأمان نفسها، مثل قوائم الكلمات المسموح بها ومهلات التنفيذ وفرض RLS. ويمكنك أيضًا استخدام SQL API مع أي وكيل خارجي، مثل [Claude Code](https://docs.anthropic.com/en/docs/claude-code) أو OpenAI Codex أو سكربتات خاصة أو Zapier webhooks. امنح الوكيل مفتاح `ebt_` الخاص بك، ووجّهه إلى نقطة النهاية الصحيحة، وسيحصل على صلاحيات القراءة والكتابة داخل حدود مساحة عملك فقط.

## مزايا متتبّع الميزانية

هذا ليس مجرد دفتر مصروفات بسيط، بل أداة تحتوي على ما تتوقعه من منتج تجاري ناضج:

- **شبكة ميزانية**: الصفوف تمثل الفئات، والأعمدة تمثل الأشهر. الأشهر الماضية تعرض الأرقام الفعلية، والأشهر المقبلة تعرض التوقعات. يمكنك التخطيط لـ 12 شهرًا مسبقًا ورؤية الأرصدة المتوقعة بسرعة.
- **دعم متعدد العملات**: كل معاملة تُحفظ بعملتها الأصلية. تُجلب أسعار الصرف اليومية تلقائيًا من ECB وCBR وNBS، ويُجرى التحويل إلى عملة التقارير وقت الاستعلام عبر SQL، من دون أعمدة محسوبة مسبقًا ومن دون فقدان في الدقة.
- **أرصدة الحسابات**: تتبع الحسابات الجارية، والادخار، وبطاقات الائتمان، والنقد، والاستثمارات. ولكل حساب رصيد جارٍ مشتق من دفتر الأستاذ.
- **التحويلات**: يمكنك نقل الأموال بين حساباتك، حتى عبر عملات مختلفة. ويُسجَّل ذلك كقيدين في `ledger_entries` يحملان `event_id` نفسه، أحدهما سالب والآخر موجب.
- **تصنيف المعاملات**: تحدد فئاتك بنفسك من دون تصنيف جاهز مفروض عليك. والذكاء الاصطناعي يتعلم من الفئات الموجودة في بياناتك.
- **واجهة متعددة اللغات**: الإنجليزية والإسبانية والصينية والعربية والعبرية والفارسية والأوكرانية والروسية، مع دعم كامل للاتجاه من اليمين إلى اليسار.
- **عزل مساحات العمل**: يضمن Postgres Row Level Security فصل بيانات كل مستخدم بالكامل. وحتى إذا شارك عدة مستخدمين خادم قاعدة البيانات نفسه، فلن يرى أحد بيانات الآخر.
- **وضع تجريبي**: يمكنك تفعيل زر في الواجهة للانتقال إلى بيانات تجريبية داخل الذاكرة، من دون الحاجة إلى قاعدة بيانات لاستكشاف المنتج.

## مخطط Postgres مناسب للمطورين

المخطط هنا من النوع الذي يمكنك استيعابه بسرعة:

- `ledger_entries`: صف واحد لكل حركة حساب، وهو الجدول الأساسي.
- `budget_lines`: سجل ميزانية بإضافات متتابعة، وتُعتمد آخر كتابة لكل خلية.
- `budget_comments`: ملاحظات مرتبطة بخلايا الميزانية.
- `exchange_rates`: أسعار صرف يومية عامة لا تخضع لتحكم الوصول.
- `workspace_settings`: عملة التقارير الخاصة بكل مساحة عمل.
- `account_metadata`: تصنيف السيولة للحسابات.
- `accounts`: `VIEW` مشتقة من `ledger_entries`.

لا يوجد ORM، ولا إطار ترحيلات معقد. توجد فقط ملفات SQL مرقمة داخل `db/migrations/` يطبقها سكربت shell. يمكنك قراءة كل ترحيل، وفهم كل جدول، وكتابة استعلاماتك مباشرة على المخطط.

وتُدار تغييرات المخطط عبر الترحيلات. دور قاعدة البيانات `app`، الذي تستخدمه واجهة الويب، يملك صلاحيات محدودة، فلا يستطيع إنشاء الجداول أو تعديل المخطط. أما الدور `tracker`، والمستخدم فقط من سكربت الترحيل، فهو المسؤول عن أوامر DDL. هذا النوع من فصل المسؤوليات هو ما تتوقعه من نظام جاد في بيئة إنتاجية.

## لماذا يفضّل المطورون استضافة بياناتهم المالية بأنفسهم

أنت تملك أصلًا المهارات اللازمة لتشغيل هذا النظام. أنت تعرف Docker وPostgres وAWS، أو أي سحابة أخرى تفضّلها. والسؤال الحقيقي ليس إن كنت قادرًا، بل إن كانت الفائدة تستحق الجهد.

**ملكية كاملة للبيانات**: بياناتك المالية الشخصية لا تغادر بنيتك. لا تتأثر باختراقات الأطراف الثالثة، ولا تحتاج إلى قراءة سياسات خصوصية طويلة، ولا يوجد من يحوّل عادات إنفاقك إلى مادة دعائية.

**قابلية عالية للتخصيص**: يمكنك إضافة أعمدة إلى المخطط، وبناء تقاريرك الخاصة باستخدام SQL مباشرة، وربط النظام بأدواتك الحالية. تريد بوت Telegram يرسل ملخص الإنفاق اليومي؟ اكتب سكربتًا يستدعي SQL API. تريد عرض البيانات في Grafana؟ اربطها مباشرة بقاعدة Postgres. الشيفرة لديك، والتعديل بيدك.

**لا ارتهان لمزوّد واحد**: إذا قررت التوقف عن استخدام واجهة الويب، فبياناتك ستبقى محفوظة في قاعدة Postgres قياسية. يمكنك تصديرها عبر `pg_dump`، أو استعراضها بأي عميل SQL، أو نقلها لاحقًا إلى أي نظام آخر.

**فرصة للتعلّم**: قاعدة الشيفرة تقدم مثالًا عمليًا على Next.js وPostgres وDocker وAWS CDK وRow Level Security ومصادقة مفاتيح API وتكامل أدوات الذكاء الاصطناعي. وإذا كنت تبني SaaS خاصًا بك، فستجد هنا أفكارًا وأنماطًا تستحق الاقتباس.

## ابدأ الآن مع متتبّع الميزانية المفتوح المصدر

```bash
git clone https://github.com/kirill-markin/expense-budget-tracker.git
cd expense-budget-tracker
make up
```

افتح `http://localhost:3000`، وأدخل أول معاملة لديك، ثم اضبط ميزانية للشهر الحالي. وإذا أردت التجربة من دون قاعدة بيانات، فانقر زر Demo في الترويسة للانتقال إلى بيانات تجريبية داخل الذاكرة.

وعندما تصبح جاهزًا لبيئة إنتاجية، اتبع [دليل النشر على AWS](https://github.com/kirill-markin/expense-budget-tracker/blob/main/infra/aws/README.md)، أو عدّل إعداد Docker Compose بما يناسب بنيتك الخاصة.

المستودع موجود على [github.com/kirill-markin/expense-budget-tracker](https://github.com/kirill-markin/expense-budget-tracker). يمكنك وضع نجمة له، أو نسخه، أو الاكتفاء بقراءة الشيفرة. وهو مرخّص تحت MIT، لذا استخدمه بالطريقة التي تريدها.

إذا كنت تدير خوادم وقواعد بيانات في عملك أصلًا، فاستضافة المكدس نفسه لبياناتك المالية الشخصية ليست قفزة كبيرة، لكنها تمنحك في المقابل تحكمًا كاملًا في بياناتك.
