## Огляд системи

```
Browser UI  -->  Next.js web app  -->  Postgres (RLS)
                        |                  ^
                        v                  |
                 Auth service -------------+
                        ^
                        |
             Machine clients via API Gateway
                        ^
                        |
                Worker (FX fetchers) ------
```

П'ять компонентів, одна база даних:

1. **web** — застосунок Next.js з інтерфейсом і API-маршрутами
2. **auth** — вхід через одноразовий код з email і початкове підключення агентів на домені автентифікації
3. **sql-api** — AWS Lambda за API Gateway для машинних клієнтів
4. **worker** — щодня отримує обмінні курси від ECB, CBR і NBS
5. **Postgres** — єдине джерело істини з безпекою на рівні рядків

## Модель даних

- **ledger_entries** — один рядок на кожен рух коштів за рахунком: дохід, витрата або переказ
- **budget_lines** — комірки бюджету з журналом додавання; під час читання перемагає останній запис
- **budget_comments** — примітки з журналом додавання, прив'язані до комірок бюджету
- **workspace_settings** — валюта звітності та налаштування рівня робочого простору
- **account_metadata** — метадані окремих рахунків, наприклад класифікація ліквідності
- **exchange_rates** — щоденні валютні курси, які використовуються для конвертації під час запиту
- **workspaces** / **workspace_members** — ізоляція між орендарями
- **accounts** — подання, побудоване на основі записів у книзі операцій

## Мультивалютність

Усі суми зберігаються у вихідній валюті. Конвертація у валюту звітності відбувається під час читання через SQL-з'єднання з `exchange_rates`. Жодної попередньої конвертації з втратою точності.

## Автентифікація

Два режими через змінну середовища `AUTH_MODE`:

- `none` — без автентифікації, один локальний робочий простір
- `cognito` — безпарольний вхід через одноразовий код з email у AWS Cognito, з відкритою реєстрацією

Для машинних клієнтів публічна точка виявлення — `GET /v1/`. Початкове підключення агентів використовує одноразовий код з email на домені автентифікації, повертає довгоживучий `ApiKey`, а далі виконує SQL через машинний API за API Gateway.

Кожен користувач отримує ізольований робочий простір. Політики RLS перевіряють членство в ньому на кожному запиті, зокрема для SQL-запитів із машинного інтерфейсу.
