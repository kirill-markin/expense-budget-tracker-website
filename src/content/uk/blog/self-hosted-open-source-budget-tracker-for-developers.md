Якщо ви розробник, ваші фінансові дані, швидше за все, живуть на чиємусь чужому сервері. Кожен budget tracker і expense tracker — Mint, YNAB, Copilot, Lunch Money — зберігає ваші транзакції, баланси й патерни витрат у своїй хмарі. Ви просто довіряєте їм, що їх не зламають, що вони не продадуть ваші дані й що не закриються (RIP Mint, 2024).

Є кращий варіант, якщо вам комфортні Docker і Postgres: розгорнути self-hosted open-source трекер бюджету й тримати все на власній інфраструктурі.

## Трекер витрат з відкритим кодом, який ви розгортаєте самі

[Expense Budget Tracker](https://github.com/kirill-markin/expense-budget-tracker) — це повністю open-source система обліку витрат і бюджету на Postgres. Ви клонуєте репозиторій, запускаєте `make up` і отримуєте робочий застосунок на `localhost:3000` із реальною базою, яку контролюєте.

Без створення акаунта, без вивантаження даних із вашої машини, без абонплати. Ліцензія MIT — форкайте, змінюйте, робіть що завгодно.

![Budget table showing past actuals, current month tracking, and future monthly forecast by category](/blog/budget-view-example.jpg)

Стек тут прямий: Next.js для веб-інтерфейсу, Postgres 18 для зберігання й TypeScript-воркер, який щодня підтягує валютні курси. Усе працює в Docker-контейнерах через один `docker-compose.yml`.

## Розгортайте через Docker або в AWS

У репозиторії з коробки є два варіанти розгортання:

**Local Docker Compose** — чотири команди, і все працює:

```bash
git clone https://github.com/kirill-markin/expense-budget-tracker.git
cd expense-budget-tracker
open -a Docker   # start Docker if not running
make up          # Postgres + migrations + web + worker
```

Відкриваєте `http://localhost:3000` і починаєте вводити транзакції. Дані Postgres зберігаються у Docker volume. Ось і весь сетап.

**AWS CDK** — повноцінне production-розгортання одним скриптом:

```bash
bash scripts/bootstrap.sh --region eu-central-1
```

Це піднімає ECS Fargate, RDS Postgres, ALB із HTTPS, Cognito для auth, WAF, моніторинг у CloudWatch, автоматичні бекапи й CI/CD через GitHub Actions. Орієнтовна вартість — близько 50 доларів на місяць, і ви отримуєте інфраструктуру enterprise-рівня, якою повністю володієте. У [гайді з розгортання](https://github.com/kirill-markin/expense-budget-tracker/blob/main/infra/aws/README.md) розписано кожен крок — від створення AWS-акаунта до налаштування DNS у Cloudflare.

Оскільки під капотом це просто Postgres + Docker, ви можете розгорнути застосунок де завгодно. DigitalOcean, Hetzner, Raspberry Pi в шафі, Kubernetes-кластер вашої компанії — якщо там працюють Docker і Postgres, значить, працюватиме і це.

## Кінцева точка SQL API для програмного доступу

Більшість бюджетних застосунків дає вам веб-інтерфейс і більше нічого. Тут є **SQL Query API** поверх HTTP — endpoint `POST /v1/sql`, який приймає сирі SQL-запити й повертає JSON.

```bash
curl -X POST https://api.your-domain.com/v1/sql \
  -H "Authorization: ApiKey ebta_a7Bk9mNp..." \
  -H "X-Workspace-Id: workspace-id" \
  -H "Content-Type: application/json" \
  -d '{"sql": "SELECT category, SUM(amount) AS total FROM ledger_entries WHERE kind = '\''spend'\'' AND ts >= DATE_TRUNC('\''month'\'', CURRENT_DATE) GROUP BY category ORDER BY total"}'
```

Ви створюєте API-ключ у Settings, обираєте target workspace ID — і будь-який HTTP-клієнт може запитувати ваші дані. Це простий REST endpoint — без GraphQL, без ORM-абстракцій, без SDK, який ще треба вивчити. Просто SQL на вході, JSON на виході.

Модель безпеки тут сувора: API-ключі зберігаються як SHA-256-хеші (plaintext ніколи не зберігається), запити обмежені `SELECT`/`INSERT`/`UPDATE`/`DELETE` (без DDL), є таймаут на 30 секунд, ліміт у 100 рядків на відповідь і rate limiting 10 запитів на секунду на ключ. Усі запити проходять через Postgres Row Level Security — ту саму ізоляцію, яку використовує вебзастосунок — тому API-ключ бачить лише дані у workspace свого власника.

## Побудовано для AI-агентів і LLM

Саме SQL API робить AI-інтеграцію практичною для особистих фінансів — AI-агенту потрібен прямий доступ до бази, щоб читати й писати фінансові дані.

Подумайте, як більшість людей працює з AI-асистентами сьогодні. Вони вставляють скриншот банківської виписки в Claude або ChatGPT, просять категоризувати витрати — і отримують милий текстовий summary. Потім вручну копіюють числа в якийсь інструмент. Це флоу з 2023 року.

Коли є SQL API, AI-агент не просто аналізує дані — він **пише у вашу базу**. Флоу виглядає так:

1. Ви кидаєте банківську виписку (CSV, PDF або скриншот) AI-агенту
2. Агент читає кожну транзакцію, звіряє категорії з тими, що вже існують, і робить `INSERT` у `ledger_entries`
3. Агент перевіряє баланс рахунку проти цифри з банку
4. Ви витрачаєте 5 хвилин на перегляд замість години на ручне введення

Схема бази спроєктована саме під це. Сім плоских таблиць, без вкладеного JSON, без складних join'ів для базових операцій. Таблиця `ledger_entries` навмисно проста: один рядок на один рух по рахунку, з чіткими назвами колонок. LLM може написати коректний `INSERT` з першої спроби, бо тут просто нема в чому заплутатися.

У Expense Budget Tracker також є **вбудований AI-чат** у веб-інтерфейсі. Підключаєте свій ключ OpenAI або Anthropic — і отримуєте асистента з інструментом `query_database`, який може робити `SELECT`, `INSERT`, `UPDATE` і `DELETE` прямо у вашому Postgres. Завантажуєте скриншот із банківського застосунку — AI парсить усі транзакції, просить підтвердження й вставляє їх. Він дотримується суворого протоколу: спочатку знаходить ваші існуючі категорії, перевіряє дублікати, звіряє баланси і пише лише після вашої явної згоди.

AI-чат підтримує моделі Claude (Anthropic) і GPT (OpenAI). Обидві використовують той самий database tool із тими самими правилами безпеки — whitelist по ключових словах, statement timeouts і примусове RLS. Ви також можете використовувати SQL API із будь-яким зовнішнім агентом: [Claude Code](https://docs.anthropic.com/en/docs/claude-code), OpenAI Codex, власними скриптами або Zapier webhooks. Передаєте агенту свій `ebt_`-ключ, endpoint — і він отримує повний read/write-доступ у межах вашого workspace.

## Можливості бюджетного трекера

Це не мінімалістичний ledger для витрат. Тут є все, чого ви очікуєте від комерційного продукту:

- **Budget grid** — рядки це категорії, колонки це місяці. Минулі місяці показують actuals, майбутні — ваш forecast. Можна планувати на 12 місяців уперед і бачити прогнозовані баланси
- **Мультивалютна підтримка** — кожна транзакція зберігається в рідній валюті. Щоденні курси з ECB, CBR і NBS підтягуються автоматично. Конвертація у вашу валюту звітності відбувається під час запиту через SQL join'и — без precomputed columns і без втрати точності
- **Баланси рахунків** — checking, savings, кредитки, готівка, інвестиції. Кожен рахунок має поточний баланс, виведений із ledger
- **Перекази** — рух грошей між вашими рахунками, включно з різними валютами. Два записи в ledger з однаковим `event_id`, один негативний, один позитивний
- **Категоризація транзакцій** — вільні категорії, які задаєте ви. Ніякої примусової таксономії. AI вчиться на ваших поточних категоріях
- **Мультимовний UI** — англійська, іспанська, китайська, арабська, іврит, фарсі, українська, російська. Повна підтримка RTL
- **Workspace isolation** — Postgres Row Level Security гарантує повну ізоляцію даних кожного користувача. Навіть на спільному сервері користувачі не бачать чужих даних
- **Demo mode** — можна перемкнутися на in-memory demo data просто з UI. Без бази, щоб лише подивитися інтерфейс

## Postgres-схема, зроблена для розробників

Уся схема вміщується в голові:

- `ledger_entries` — один рядок на один рух по рахунку
- `budget_lines` — append-only план бюджету, де для клітинки працює last-write-wins
- `budget_comments` — нотатки до клітинок бюджету
- `exchange_rates` — щоденні FX-курси
- `workspace_settings` — валюта звітності по workspace
- `account_metadata` — класифікація ліквідності
- `accounts` — VIEW, похідний від `ledger_entries`

Жодного ORM. Жодного migration framework. Просто пронумеровані SQL-файли в `db/migrations/`, які застосовує shell-скрипт. Ви можете прочитати кожну міграцію, зрозуміти кожну таблицю й писати запити напряму до схеми.

Зміни схеми йдуть через міграції. Роль бази `app` (яку використовує вебзастосунок) має обмежені права — не може створювати таблиці чи змінювати схему. Роль `tracker` (яку використовує лише migration-скрипт) виконує DDL. Саме такого поділу відповідальностей і чекаєш від production-системи.

## Чому розробники розгортають свої фінансові дані самі

У вас уже є потрібні навички. Ви розумієте Docker, Postgres і AWS або будь-яку іншу хмару. Питання лише в тому, чи виправдовують вигоди витрачені зусилля.

**Повне володіння даними** — ваші персональні фінансові дані не залишають вашу інфраструктуру. Жодні сторонні витоки вас не зачіпають. Ніяких політик приватності, які треба перечитувати. Ніякої аналітики по ваших витратах для рекламодавців.

**Кастомізація** — додавайте колонки в схему, пишіть власні звіти на сирому SQL, підключайте систему до своїх інструментів. Хочете Telegram-бота, який щодня надсилає summary витрат? Напишіть скрипт поверх SQL API. Хочете візуалізувати дані в Grafana? Підключіть її до Postgres. Код ваш — міняйте як хочете.

**Жодного vendor lock-in** — якщо перестанете користуватися веб-інтерфейсом, дані все одно лишаються у стандартній Postgres-базі. Їх можна експортувати через `pg_dump`, читати з будь-якого SQL-клієнта або мігрувати кудись іще.

**Навчальна цінність** — це реальний приклад Next.js + Postgres + Docker + AWS CDK + Row Level Security + API key auth + AI tool integration. Якщо ви будуєте власний SaaS, тут є що піддивитися.

```bash
make up
```

Відкрийте `http://localhost:3000`. Запишіть першу транзакцію. Налаштуйте бюджет на поточний місяць. Якщо хочете спочатку просто подивитися інтерфейс без бази, натисніть Demo в хедері, щоб переключитися на in-memory sample data.
