Если вы разработчик, ваши финансовые данные, скорее всего, живут на чужом сервере. Любой трекер бюджета и расходов — Mint, YNAB, Copilot, Lunch Money — хранит ваши транзакции, балансы и паттерны расходов у себя в облаке. Вы просто надеетесь, что их не взломают, что они не продадут ваши данные и что продукт однажды не закроют.

Есть более хороший вариант, если вам комфортны Docker и Postgres: поднять open source трекер бюджета у себя и держать все на собственной инфраструктуре.

## Open source трекер расходов, который вы разворачиваете сами

[Expense Budget Tracker](https://github.com/kirill-markin/expense-budget-tracker) — это полностью open source система для учета расходов и бюджета на Postgres. Вы клонируете репозиторий, запускаете `make up` и получаете рабочее приложение на `localhost:3000` с реальной базой, которой управляете вы.

Никакой регистрации, никакого ухода данных с вашей машины и никаких подписок. Лицензия MIT — форкайте, меняйте, делайте что хотите.

![Budget table showing past actuals, current month tracking, and future monthly forecast by category](/blog/budget-view-example.jpg)

Стек прямолинейный: Next.js для веб-интерфейса, Postgres 18 для хранения и TypeScript worker, который подтягивает ежедневные курсы валют. Все крутится в Docker-контейнерах через один `docker-compose.yml`.

## Self-host через Docker или деплой в AWS

В репозитории уже из коробки есть два сценария деплоя:

**Локальный Docker Compose** — четыре команды, и система работает:

```bash
git clone https://github.com/kirill-markin/expense-budget-tracker.git
cd expense-budget-tracker
open -a Docker   # start Docker if not running
make up          # Postgres + migrations + web + worker
```

Открываете `http://localhost:3000` и начинаете вносить транзакции. Данные Postgres живут в Docker volume. На этом вся настройка.

**AWS CDK** — полноценный production-деплой одним скриптом:

```bash
bash scripts/bootstrap.sh --region eu-central-1
```

Это поднимает ECS Fargate, RDS Postgres, ALB с HTTPS, Cognito для auth, WAF, мониторинг CloudWatch, автоматические бэкапы и CI/CD через GitHub Actions. Оценочная стоимость — около $50 в месяц, и вы получаете инфраструктуру уровня enterprise, которой полностью владеете. [Руководство по деплою](https://github.com/kirill-markin/expense-budget-tracker/blob/main/infra/aws/README.md) проходит по всем шагам — от создания AWS-аккаунта до настройки DNS в Cloudflare.

Поскольку это всего лишь Postgres и Docker, вы можете разместить систему и где угодно еще. DigitalOcean, Hetzner, Raspberry Pi в шкафу, Kubernetes-кластер вашей компании — если там крутятся Docker и Postgres, это запустится.

## SQL API для программного доступа

Большинство бюджетных приложений дает вам веб-интерфейс и больше ничего. Здесь есть **SQL Query API** поверх HTTP — endpoint `POST /v1/sql`, который принимает сырой SQL и возвращает JSON.

```bash
curl -X POST https://api.your-domain.com/v1/sql \
  -H "Authorization: ApiKey ebta_a7Bk9mNp..." \
  -H "X-Workspace-Id: workspace-id" \
  -H "Content-Type: application/json" \
  -d '{"sql": "SELECT category, SUM(amount) AS total FROM ledger_entries WHERE kind = '\''spend'\'' AND ts >= DATE_TRUNC('\''month'\'', CURRENT_DATE) GROUP BY category ORDER BY total"}'
```

Вы создаете API-ключ в Settings, выбираете нужный workspace ID — и любой HTTP-клиент уже может читать ваши данные. Это простой REST endpoint: никакого GraphQL, никакой ORM-магии, никаких SDK, которые нужно учить. Просто SQL на входе, JSON на выходе.

Модель безопасности строгая: API-ключи хранятся как SHA-256-хэши, разрешены только `SELECT/INSERT/UPDATE/DELETE`, действует таймаут 30 секунд, лимит 100 строк на ответ и rate limit 10 запросов в секунду на ключ. Все запросы проходят через Postgres Row Level Security — тот же механизм изоляции, который использует веб-приложение, — поэтому API-ключ видит только данные своего workspace.

## Система построена под AI-агентов и LLM

Именно SQL API делает AI-интеграцию в личных финансах практичной — агенту нужен прямой доступ к базе, чтобы читать и писать финансовые данные.

Подумайте, как вы сегодня взаимодействуете с AI. Вы вставляете скриншот банковской выписки в Claude или ChatGPT, просите категоризировать расходы, получаете красивую текстовую сводку. Потом вручную переносите эти цифры туда, где реально ведете деньги. Это сценарий из 2023 года.

Когда есть SQL API, AI-агент не просто анализирует данные — он **пишет в вашу базу**. Workflow становится таким:

1. Вы бросаете банковскую выписку в CSV, PDF или даже скриншот в AI-агента
2. Агент читает каждую транзакцию, сопоставляет категории с уже существующими и делает `INSERT` в `ledger_entries`
3. Агент проверяет баланс счета против цифры в банке
4. Вы тратите 5 минут на ревью вместо часа ручного ввода

Схема базы спроектирована под это. Семь плоских таблиц, без вложенного JSON и без сложных join'ов для базовых операций. Таблица `ledger_entries` нарочно проста — одна строка на каждое движение по счету и понятные названия колонок. LLM умеет писать корректные INSERT-запросы с первого раза именно потому, что здесь почти не на чем запутаться.

У Expense Budget Tracker есть и **встроенный AI-чат** в веб-интерфейсе. Подключаете API-ключ OpenAI или Anthropic — и получаете ассистента с инструментом `query_database`, который может делать `SELECT`, `INSERT`, `UPDATE` и `DELETE` прямо в вашей Postgres. Загружаете скриншот банковского приложения, AI разбирает каждую транзакцию, просит подтвердить и записывает. Он следует строгому протоколу: сначала изучает существующие категории, проверяет дубликаты, сверяет балансы и пишет в базу только после вашего явного подтверждения.

AI-чат поддерживает модели Claude от Anthropic и GPT от OpenAI. Обе используют один и тот же database tool с одинаковыми правилами безопасности — whitelisting ключевых слов, таймауты запросов и принудительное применение RLS. Тот же SQL API можно использовать и с любым внешним агентом: [Claude Code](https://docs.anthropic.com/en/docs/claude-code), OpenAI Codex, собственными скриптами или Zapier webhooks. Даете агенту ключ `ebt_`, указываете endpoint — и он получает полный read/write-доступ в пределах вашего workspace.

## Что умеет этот бюджетный трекер

Это не голый ledger расходов. Все функции, которых вы ждете от коммерческого продукта, уже на месте:

- **Сетка бюджета** — строки это категории, колонки это месяцы. В прошлых месяцах факт, в будущих — прогноз. Можно планировать на 12 месяцев вперед и сразу видеть прогнозные балансы
- **Мультивалютность** — каждая транзакция хранится в своей родной валюте. Курсы от ECB, CBR и NBS подтягиваются ежедневно. Конвертация в валюту отчетности происходит в момент чтения через SQL join'ы, без предрасчитанных колонок и потери точности
- **Балансы счетов** — расчетные счета, сбережения, кредитки, наличные, инвестиции. У каждого счета есть текущий баланс, вычисляемый из ledger
- **Переводы** — перемещения денег между своими счетами, включая кросс-валютные. Две записи `ledger_entries` с одним `event_id`, одна отрицательная, другая положительная
- **Категоризация транзакций** — свободные категории, которые определяете вы. Никакой навязанной таксономии. AI учится на уже существующих категориях
- **Мультиязычный интерфейс** — English, Spanish, Chinese, Arabic, Hebrew, Farsi, Ukrainian, Russian. Полная RTL-поддержка
- **Изоляция workspace'ов** — Postgres Row Level Security гарантирует полную изоляцию данных пользователей даже в общей базе
- **Demo mode** — кнопка в интерфейсе, которая переключает продукт на демо-данные в памяти без базы

## Схема Postgres, понятная разработчикам

Всю схему реально удержать в голове:

- `ledger_entries` — одна строка на каждое движение по счету
- `budget_lines` — append-only план бюджета
- `budget_comments` — заметки к ячейкам бюджета
- `exchange_rates` — ежедневные FX-курсы
- `workspace_settings` — валюта отчетности на workspace
- `account_metadata` — классификация ликвидности
- `accounts` — VIEW, построенный из `ledger_entries`

Никакой ORM. Никакого фреймворка миграций. Просто нумерованные SQL-файлы в `db/migrations/`, которые применяются shell-скриптом. Можно открыть каждую миграцию, понять каждую таблицу и писать запросы прямо по схеме.

Изменения схемы идут через миграции. Роль базы `app`, которой пользуется веб-приложение, имеет ограниченные права — она не может создавать таблицы и менять схему. Роль `tracker`, которую использует только скрипт миграций, отвечает за DDL. Именно такого разделения обязанностей и ждешь от production-системы.

## Почему разработчики разворачивают финансовые данные у себя

У вас уже есть для этого все навыки. Вы понимаете Docker, Postgres и AWS или любой другой облачный провайдер. Вопрос только в том, стоят ли преимущества этих усилий.

**Полное владение данными** — ваши финансовые данные никогда не уходят на чужую инфраструктуру. Никакие утечки третьих сторон вас не затрагивают. Никаких приватных политик, которые надо читать. Никакой аналитики по вашим тратам для рекламодателей.

**Кастомизация** — можно добавлять колонки в схему, строить кастомные отчеты на чистом SQL, подключать систему к уже существующим инструментам. Хотите Telegram-бота с ежедневным отчетом по тратам? Пишете скрипт, который дергает SQL API. Хотите визуализацию в Grafana? Подключаете ее к Postgres. Код ваш, меняйте как хотите.

**Отсутствие привязки к вендору** — даже если перестанете пользоваться веб-интерфейсом, данные останутся в стандартной базе Postgres. Можно выгрузить их через `pg_dump`, читать из любого SQL-клиента или мигрировать куда угодно.

**Обучение** — кодовая база дает реальный пример Next.js + Postgres + Docker + AWS CDK + Row Level Security + API-key auth + интеграции с AI-tools. Если вы строите SaaS, там много паттернов, которые стоит украсть.

## С чего начать с open source трекером бюджета

```bash
git clone https://github.com/kirill-markin/expense-budget-tracker.git
cd expense-budget-tracker
make up
```

Откройте `http://localhost:3000`. Добавьте первую транзакцию. Настройте бюджет на текущий месяц. Если хотите просто посмотреть интерфейс без базы, нажмите кнопку Demo в шапке — она переключит продукт на пример в памяти.

Когда будете готовы к production, идите по [руководству по деплою в AWS](https://github.com/kirill-markin/expense-budget-tracker/blob/main/infra/aws/README.md) или адаптируйте Docker Compose под свою инфраструктуру.

Репозиторий находится на [github.com/kirill-markin/expense-budget-tracker](https://github.com/kirill-markin/expense-budget-tracker). Поставьте звезду, сделайте fork или просто читайте код. Лицензия MIT — используйте как хотите.

Если вы и так управляете серверами и базами по работе, поднять такой же стек для личных финансов — небольшой шаг. А взамен вы получаете полный контроль над данными.
