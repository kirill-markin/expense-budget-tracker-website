احتمالاً همین حالا هم به نوعی از AI برای امور مالی شخصی استفاده می‌کنید. شاید صورت‌حساب بانکی را در ChatGPT کپی می‌کنید و از آن می‌خواهید خرج‌ها را دسته‌بندی کند. یا از اپ بانکی‌تان اسکرین‌شات می‌گیرید و از Claude می‌پرسید این ماه چقدر برای خرید مواد غذایی خرج کرده‌اید.

این یک بار جواب می‌دهد. اما پاسخ داخل همان چت می‌ماند. چیزی ذخیره نمی‌شود، چیزی پیگیری نمی‌شود و هفتهٔ بعد دوباره همان کار را تکرار می‌کنید. AI داده‌های شما را می‌خواند، یک خلاصه می‌دهد و بعد همه‌چیز تمام می‌شود.

یک راه مفیدتر برای استفاده از AI در پیگیری هزینه وجود دارد. به‌جای اینکه از AI بخواهید اسکرین‌شات‌ها را تحلیل کند، به آن دسترسی واقعیِ نوشتن در دیتابیس مالی‌تان بدهید. بگذارید AI تراکنش‌ها را ثبت کند، بودجه را به‌روزرسانی کند و مانده‌حساب‌ها را مستقیم بررسی کند؛ نه اینکه فقط درباره‌شان حرف بزند.

## «پیگیری هزینه با AI» در عمل واقعاً چه شکلی دارد

Kirill Markin، سازندهٔ [Expense Budget Tracker](https://expense-budget-tracker.com/fa/)، بیش از پنج سال است که هر تراکنش شخصی را دسته‌بندی می‌کند. اول این کار را دستی انجام می‌داد و بعد شروع کرد به ساختن ابزارهایی که سریع‌ترش کنند. سیستم فعلی از یک عامل AI استفاده می‌کند که مستقیم از طریق SQL API به دیتابیس وصل می‌شود.

روتین هفتگی او این‌طور است: صورت‌حساب‌های بانکی را دانلود می‌کند (CSV یا PDF)، آن‌ها را داخل یک عامل AI می‌اندازد و می‌گذارد عامل هر تراکنش را parse و ثبت کند. عامل از روی ورودی‌های قبلی دسته‌بندی‌های هزینه را می‌شناسد، بنابراین بیشتر تراکنش‌ها را به‌تنهایی درست تطبیق می‌دهد. Kirill نتیجهٔ کار AI را مرور می‌کند، چند اشتباه کوچک را اصلاح می‌کند و تمام. کل فرایند حدود ۱۰ دقیقه طول می‌کشد، در حالی که وقتی همه‌چیز را دستی وارد می‌کرد حدود یک ساعت زمان می‌برد.

همین رویکرد با [Claude Code](https://docs.anthropic.com/en/docs/claude-code)، [OpenAI Codex](https://openai.com/index/codex/)، custom GPTها یا هر عامل AI دیگری که بتواند endpointهای HTTP را صدا بزند کار می‌کند. مادهٔ اصلیِ لازم دسترسی مستقیم به دیتابیس است؛ نه plugin، نه افزونهٔ مرورگر، بلکه یک API key که به AI اجازه بدهد داده‌های مالی شما را بخواند و بنویسد.

## چطور عامل AI را به امور مالی‌تان وصل کنید

[Expense Budget Tracker](https://expense-budget-tracker.com/fa/) یک سیستم مالی شخصی متن‌باز است که روی Postgres ساخته شده. این سیستم یک SQL API در آدرس `POST /v1/sql` دارد که queryهای SQL را از طریق HTTP می‌گیرد و نتیجه را به‌صورت JSON برمی‌گرداند.

برای وصل‌کردن هر عامل AI:

1. اپ را باز کنید و به **Settings → API Keys → Create key** بروید
2. کلید را کپی کنید (با `ebt_` شروع می‌شود و فقط یک بار آن را می‌بینید)
3. دو چیز را به عامل AI بگویید: URL endpoint و خودِ کلید

همین. حالا عامل می‌تواند داده‌های هزینهٔ شما را query و ویرایش کند. نه MCP server لازم است، نه plugin، نه integration سفارشی برای نگهداری. هر AIای که بتواند یک درخواست HTTP POST بفرستد، همان لحظه آماده است.

```bash
curl -X POST https://api.expense-budget-tracker.com/v1/sql \
  -H "Authorization: ApiKey ebta_your_key_here" \
  -H "X-Workspace-Id: workspace-id" \
  -H "Content-Type: application/json" \
  -d '{"sql": "SELECT * FROM ledger_entries ORDER BY ts DESC LIMIT 10"}'
```

پاسخ به شکل یک آرایهٔ JSON از ردیف‌ها برمی‌گردد. نه pagination token، نه objectهای تو‌در‌تو، نه SDK.

## عامل AI شما با این دسترسی چه کارهایی می‌تواند انجام دهد

با API key، عامل AI روی دادهٔ واقعیِ هزینه و بودجهٔ شما کار می‌کند؛ نه روی یک کپی، نه یک خلاصه، بلکه روی دیتابیس زنده:

**هزینه‌ها را parse و ثبت می‌کند.** صورت‌حساب بانکی (CSV، PDF یا اسکرین‌شات اپ بانکی) را داخل عامل AI بیندازید. عامل هر خط را می‌خواند، مبلغ، تاریخ، طرف حساب و دسته‌بندی را تشخیص می‌دهد و بعد یک دستور INSERT در جدول `ledger_entries` می‌نویسد. هر هزینه مستقیم وارد دیتابیس می‌شود.

**تراکنش‌ها را با استفاده از دسته‌بندی‌های موجودتان دسته‌بندی می‌کند.** عامل اول query می‌کند که شما از قبل چه دسته‌بندی‌هایی استفاده می‌کنید. اگر چند ماه است «Whole Foods» را به‌عنوان «groceries» ثبت می‌کنید، AI این الگو را می‌فهمد و همان را ادامه می‌دهد. لازم نیست هر بار دوباره سیستم‌تان را توضیح بدهید.

**مانده‌حساب‌ها را بررسی می‌کند.** بعد از ثبت همهٔ هزینه‌های یک صورت‌حساب بانکی، عامل می‌تواند روی view `accounts` query بزند و جمع‌ها را با عددهای خودِ بانک مقایسه کند. اگر چیزی جور درنیاید، می‌فهمید یک تراکنش جا افتاده است.

**پیش‌بینی بودجه را به‌روزرسانی می‌کند.** جدول `budget_lines` برنامهٔ ماهانهٔ بودجه را نگه می‌دارد؛ درآمد مورد انتظار و هزینهٔ برنامه‌ریزی‌شده به‌ازای هر دسته‌بندی. عامل AI می‌تواند actualهای این ماه را بخواند، با برنامه مقایسه کند و برای ماه بعد تغییر پیشنهاد بدهد یا مستقیم اعمالش کند.

**با چند ارز کار می‌کند.** هر تراکنش در دیتابیس در ارز اصلی خودش باقی می‌ماند. نرخ‌های تبدیل هر روز از ECB، CBR و NBS دریافت می‌شوند. AI لازم نیست چیزی را تبدیل کند؛ دیتابیس خودش محاسبات ارزی را هنگام query انجام می‌دهد.

schema عمداً تخت و ساده نگه داشته شده: هفت جدول، نام ستون‌های روشن، بدون ساختارهای عمیق و تو‌در‌تو. مدل‌های AI روی این schema از همان بار اول SQL درست تولید می‌کنند چون تقریباً هیچ چیز برای بدفهمی وجود ندارد.

## چت AI داخلی برای کارهای سریع

Expense Budget Tracker یک دستیار AI هم داخل خودِ رابط وب دارد. API key مربوط به OpenAI یا Anthropic را در Settings وصل می‌کنید و چت به یک ابزار `query_database` دسترسی پیدا می‌کند؛ همان دسترسی SQL، اما از داخل خودِ اپ.

این برای کارهای سریع راحت است: یک اسکرین‌شات از رسید آپلود کنید، از AI بخواهید آن را به‌عنوان هزینه ثبت کند، تأیید کنید، تمام. AI داخلی از یک پروتکل سخت‌گیرانه پیروی می‌کند: دسته‌بندی‌های موجود شما را بررسی می‌کند، دنبال تراکنش‌های تکراری می‌گردد، مانده‌حساب‌ها را تطبیق می‌دهد و فقط بعد از تأیید شما چیزی را در دیتابیس می‌نویسد.

برای کارهای بزرگ‌تر، مثل پردازش دسته‌ای چند صورت‌حساب بانکی، ساختن گردش‌کار خودکار یا اتصال به سیستم‌های دیگر، SQL API بیرونی عملی‌تر است. می‌توانید از هر عامل یا script بیرون از اپ به آن وصل شوید.

## چرا SQL مستقیم از MCP server و plugin بهتر است

الان MCP serverها، actionهای custom GPT و pluginهای وابسته به provider خیلی محبوب شده‌اند برای وصل‌کردن AI به ابزارهای بیرونی. برای امور مالی شخصی، فقط قطعات متحرکِ اضافه وارد ماجرا می‌کنند.

MCP server یک فرایند اضافه است که باید اجرا و زنده نگهش دارید. اگر کرش کند، AI وسط گفت‌وگو دسترسی‌اش به داده‌های هزینه قطع می‌شود. pluginهای custom GPT فقط داخل ChatGPT کار می‌کنند؛ اگر به Claude بروید یا عامل خودتان را بسازید به دردتان نمی‌خورند. integrationهای وابسته به provider هم هر بار که provider API را عوض کند می‌شکنند.

SQL API همهٔ این‌ها را حذف می‌کند. رابط فقط یک endpoint HTTP و زبان SQL است. هر دو دهه‌هاست وجود دارند و جایی نمی‌روند. از یک مدل AI به مدل دیگر بروید؛ همان API key، همان endpoint، همان SQL. خودِ عامل اهمیتی نمی‌دهد که داخل ChatGPT اجرا می‌شود، Claude Code است یا یک script پایتون که خودتان نوشته‌اید.

## آیا دادن دسترسی مستقیم دیتابیس به AI امن است؟

بله، اگر محدودیت‌های درست اعمال شده باشند. SQL API در Expense Budget Tracker چند لایهٔ محافظتی enforce می‌کند:

هر query از Postgres Row Level Security عبور می‌کند. API key به کاربر و فضای‌کار شما متصل است؛ AI فقط می‌تواند داده‌های هزینهٔ شما را ببیند و تغییر دهد، نه دادهٔ کسی دیگر.

فقط عملیات داده مجاز است: `SELECT`، `INSERT`، `UPDATE` و `DELETE`. عامل AI نمی‌تواند جدول بسازد، چیزی را drop کند یا مجوزها را تغییر بدهد. چند statement در یک درخواست مسدود است. `set_config()` هم بسته شده تا جلوی escalation گرفته شود.

API keyها به شکل SHA-256 hash ذخیره می‌شوند؛ متن خام کلید هرگز در دیتابیس باقی نمی‌ماند. می‌توانید هر لحظه کلید را از Settings لغو کنید. اگر عضوی را از workspace حذف کنید، همهٔ کلیدهای او هم خودکار حذف می‌شوند.

rate limit استفاده را به ۱۰ درخواست در ثانیه و ۱۰٬۰۰۰ درخواست در روز برای هر کلید محدود می‌کند. queryها بعد از ۳۰ ثانیه timeout می‌شوند. پاسخ‌ها حداکثر ۱۰۰ ردیف برمی‌گردانند. این اعداد برای پیگیری هزینه و بودجه‌بندی با AI کاملاً کافی‌اند و در عین حال جلوی رفتار runaway را می‌گیرند.

## چند نکتهٔ عملی برای پیگیری هزینه با AI

چند چیز که بر اساس استفادهٔ واقعی روزمره، workflow پیگیری هزینه با AI را نرم‌تر می‌کنند:

**نام دسته‌بندی‌های هزینه را ثابت نگه دارید.** AI از داده‌های قبلی شما یاد می‌گیرد. اگر گاهی دسته‌ای را «restaurants» بنامید و گاهی «dining out»، عامل گیج می‌شود. برای هر دسته یک نام انتخاب کنید و به آن بچسبید.

**هر هفته مانده‌حساب‌ها را تطبیق دهید.** بعد از اینکه AI هزینه‌ها را از صورت‌حساب ثبت کرد، بررسی کنید که ماندهٔ حساب در سیستم با بانک‌تان یکی باشد. این کار تراکنش‌های جاافتاده یا تکراری را زود پیدا می‌کند، قبل از اینکه روی هم انباشته شوند.

**با یک حساب شروع کنید.** از روز اول سعی نکنید همهٔ حساب‌های بانکی، کارت‌های اعتباری و حساب‌های سرمایه‌گذاری‌تان را راه‌اندازی کنید. از حساب جاری اصلی شروع کنید. بگذارید AI چند هفته همان را مدیریت کند. وقتی workflow جا افتاد، حساب‌های بیشتری اضافه کنید.

**هر بار دسته‌بندی AI را مرور کنید.** AI بیشتر تراکنش‌ها را درست می‌زند، اما گاهی چیزی را اشتباه دسته‌بندی می‌کند؛ مخصوصاً merchantهای جدید یا خرج‌های غیرمعمول را. پنج دقیقه وقت بگذارید و بررسی کنید. اصلاح اشتباه‌های AI دقت آینده را بالا می‌برد، چون دفعهٔ بعد دادهٔ اصلاح‌شده همان چیزی است که می‌بیند.

**فقط به پیگیری خرج اکتفا نکنید؛ از جدول بودجه هم استفاده کنید.** ثبت آنچه قبلاً خرج کرده‌اید مفید است اما محدود. ارزش واقعی در نگه‌داشتن یک بودجهٔ غلتان ۱۲ ماهه است؛ ردیف‌ها دسته‌بندی‌اند، ستون‌ها ماه‌ها، ماه‌های آینده پیش‌بینی شما را نگه می‌دارند. عامل‌های AI در به‌روزرسانی این پیش‌بینی‌ها بر اساس الگوی خرج واقعی خوب‌اند. بعد از مرور actualهای این ماه، از عامل بخواهید بودجهٔ ماه بعد را تنظیم کند.

## شروع کار

1. در [expense-budget-tracker.com](https://expense-budget-tracker.com/fa/) ثبت‌نام کنید (رایگان و متن‌باز) یا اپ را روی سرور خودتان [self-host](https://github.com/kirill-markin/expense-budget-tracker) کنید
2. به **Settings → API Keys → Create key** بروید و کلید را کپی کنید
3. کلید، workspace ID و endpoint یعنی `https://api.expense-budget-tracker.com/v1/sql` را به عامل AI بدهید
4. یک صورت‌حساب بانکی را داخل عامل بیندازید و از او بخواهید هزینه‌ها را parse و ثبت کند

AI schema دیتابیس را کشف می‌کند، دسته‌بندی‌های هزینهٔ شما را تطبیق می‌دهد و شروع به نوشتن تراکنش‌ها می‌کند. نتیجه را مرور کنید، هر چیزی که درست نبود اصلاح کنید و یک بودجهٔ مدیریت‌شده با AI خواهید داشت.

Kirill Markin روش شخصی خودش را با جزئیات اینجا نوشته است: [How I Use AI to Handle My Expenses from Bank Accounts and Budget](https://kirill-markin.com/articles/ai-expense-tracking-bank-accounts-budget/). پنج سال پیگیری و دسته‌بندی تک‌تک تراکنش‌ها؛ همان روشی که در این مقاله توضیح داده شده، روی پول واقعی و در چند ارز و چند کشور امتحان شده است.

این ابزار تحت مجوز MIT منتشر شده و کاملاً متن‌باز است: [github.com/kirill-markin/expense-budget-tracker](https://github.com/kirill-markin/expense-budget-tracker). می‌توانید از نسخهٔ میزبانی‌شده استفاده کنید یا خودتان آن را اجرا کنید؛ SQL API در هر دو حالت یکسان کار می‌کند.
